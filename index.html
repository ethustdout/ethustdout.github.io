Socks5

协议分析

以 proxychains curl google.com 为例

样例pcap

首先看到 1、2、3 号流量包为 tcp 三次握手。紧接着 5号包为 Socks 协议。

注：下面的阶段名只是伴随个人理解的命名，并不是协议上真实的命名

协商阶段

Socks协议的第一步由两部分组成

  1. Version
    • 版本号，socks5 就是 0x5
  2. Client Authentication Method
    • 验证方式，socks5协议支持多种校验方式
    • Authentication Method Count：客户端使用多少种校验方式
    • Method[]: 校验方式

随后 Server 回复一个 Socks 包（中间的TCP ACK就忽略了）

也是两部分组成

  1. Version
    • 版本号，和 Client 发过来的一致
  2. Accept Auth Method
    • 从 Client 的 Method[] 字段里面选取一种校验方式

本样例里面没有验证

指令阶段

首先是 Client 告诉 Server 需要代理连接的目标

  1. Version
    • 版本号
  2. Command
    • 指令，包内为 Connect，表示连接
  3. Reserved
    • 保留字段，0x00
  4. Address Type
    • 0x01: IPV4，ipv4地址
    • 0x03: Domain，域名
    • 0x04: IPV6，ipv6地址
  5. Remote name
    • 目标的域名/ipv4/ipv6
    • 如果 Address Type 字段为 Domain，那么第一个字节为域名的长度（样例中为 0x0a）。
  6. Port
    • 目标端口

Server 收到请求之后和目标 Remote name:Port尝试建立连接

而后根据连接的情况返回给 Client 响应包

  1. Version
  2. Results
    • 回复类型
    • 0x00：Success
  3. Reserved
    • 保留字段
  4. Address Type
    • 地址类型，和请求包里面一样
  5. Remote Address
    • 服务端的ip地址，由于客户端一般用不到这个，所以这个字段无所谓（写成0.0.0.0都没关系）
  6. Port
    • 服务端端口

在实验过程中，不论目标端口是否打开，服务端和目标的连接是否成功，Result 字段都返回了 Success，不知道在什么情况下会返回其他的值？

转发阶段

此时的 Server 只作为目标和 Client 的转发器
